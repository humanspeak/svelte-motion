## Findings considered and rejected (so nobody re-audits them)

- `onUpdate` motion callback (upstream `motion-dom/src/node/types.ts:270`) —
  REAL gap, highest-visibility missing motion prop, but ranked #4: single prop,
  M effort, no subsystem leverage. Promote to a plan if the user wants a 4th.
- `MotionConfig.skipAnimations` — real, S-M effort, testing-story value; ranked
  below the top 3 on user impact. Candidate for a future quick win.
- `AnimatePresence propagate` / `anchorX/anchorY` exposure / `root` (Shadow DOM)
  — real but popLayout-niche; anchorX/anchorY machinery already exists
  internally (`src/lib/utils/presence.ts:137-145`), only prop exposure missing.
- `useInstantTransition` / `useInstantLayoutTransition` / `useAnimateMini` /
  `onLayoutAnimationStart/Complete` / `MotionConfig.nonce` / `isValidProp` —
  documented-but-niche; not top-3 material.
- `dom-mini` entry point — bundle-size nicety; LazyMotion + feature bundles
  (shipped) already cover the main need.
- `presenceAffectsLayout`, `isStatic`, `transformPagePoint` — upstream marks
  internal/Framer-canvas-only; not worth porting (by-design exclusion).
- Scroll/pan/tap/focus/drag-core/`animateView`/`stagger`/motion-values
  subsystems — audited PRESENT and conformant; not gaps.
- `arc()` easing + `path` transition option, `layout="x"/"y"`, `layoutAnchor`,
  `onMeasureDragConstraints` — real small gaps; deferred as post-003 follow-ups
  (several become trivial once 003 lands).
