  landing 004 first would re-clobber whileDrag transforms.

- 004 is HIGH risk and spike-gated: its Step 2 spike findings go in the table
  below before migration proceeds.

  `feDisplacementMap` attributes from MotionValues — 002's declarative SVG

  Its reference prototype (maintainer verdict: right look, not smooth) is
  archived at `assets/ai-glow-prototype.html`; the plan encodes the four
  performance MUSTs that fix the jank.

## Findings considered and rejected (so nobody re-audits them)

- `onUpdate` motion callback (upstream `motion-dom/src/node/types.ts:270`) —
  REAL gap, highest-visibility missing motion prop, ranked just below the
  audit's top 3: single prop, M effort, no subsystem leverage. Promote to a
  plan if the user wants a 5th.
- `MotionConfig.skipAnimations` — real, S-M effort, testing-story value; ranked
  below the top items on user impact. Candidate for a future quick win.
- `AnimatePresence propagate` / `anchorX/anchorY` exposure / `root` (Shadow DOM)
  — real but popLayout-niche; anchorX/anchorY machinery already exists
  internally (`src/lib/utils/presence.ts:137-145`), only prop exposure missing.
- `useInstantTransition` / `useInstantLayoutTransition` / `useAnimateMini` /
  `onLayoutAnimationStart/Complete` / `MotionConfig.nonce` / `isValidProp` —
  documented-but-niche; not top-slot material.
- `dom-mini` entry point — bundle-size nicety; LazyMotion + feature bundles
  (shipped) already cover the main need.
- `presenceAffectsLayout`, `isStatic`, `transformPagePoint` — upstream marks
  internal/Framer-canvas-only; not worth porting (by-design exclusion).
- Scroll/pan/tap/focus/drag-core/`animateView`/`stagger`/motion-values
  subsystems — audited PRESENT and conformant; not gaps.
- `arc()` easing + `path` transition option, `layout="x"/"y"`, `layoutAnchor`,
  `onMeasureDragConstraints` — real small gaps; deferred as post-004 follow-ups
  (several become trivial once 004 lands).
- Re-typing `AugmentedMotionValue` itself as `MotionValue<T> & {…}` — rejected
  in Plan 001's root-cause analysis: the class's private `current` collides
  with the public reactive `current`; the wrapper approach avoids the catch-22.
