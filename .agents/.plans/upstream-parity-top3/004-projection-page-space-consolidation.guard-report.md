## Spirit

The plan's north star was killing the two-projection-system split and the scroll-suppression heuristic that papered over it — and the diff genuinely delivers that, not a checkbox version of it. Layout measurement now flows through the upstream motion-dom node in phase-cached page space (`measurePageRect('snapshot'|'measure')`), so a viewport scroll between reads cancels exactly instead of being suppressed after the fact; the heuristic and its window scroll listener are deleted, and the behavior the plan named as the user-visible wound — layout animations silently snapping after any scroll, offscreen elements snapping instead of animating — is fixed and pinned by e2e case 2 (formerly the encoded `test.fail()` divergence, now a plain passing test) and case 3. The legacy `ProjectionNode` is fully unwired at runtime: no construction, no measurement, no event fan-out; drag slot-pinning now consumes the upstream node's own `didUpdate` delta (mirroring `VisualElementDragControls`) painting through Plan 003's writer. Public API parity (`onProjectionUpdate`/`onLayoutMeasure`) was preserved deliberately via `calcBoxDelta`/`isDeltaZero` rather than dropped — the kind of intent-fidelity the plan's "halves per-element projection cost" framing assumed but didn't spell out. What remains of the legacy file is a types-only husk kept by two out-of-scope Reorder imports — the plan's own STOP, not an executor shortcut. One honest caveat: the spike showed the shared-element `layoutId` flows and the size-corrected FLIP still run on legacy _helper functions_ (`computeFlipTransforms`/`runFlipAnimation`) by design; consolidating those onto the upstream node's built-ins is the follow-up work the plan itself deferred (crossfade, scale correction), so "retire the legacy FLIP path" is delivered for the projection-node system, not yet for every FLIP helper — consistent with the plan's Maintenance notes, but worth saying plainly.
