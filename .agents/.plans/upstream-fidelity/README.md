## Discovered during execution (follow-up candidates)

- **First-ever `controls.start()` wipes a non-neutral idle transform before WAAPI
  captures its from-value** (found by plan 005's executor): the animation-controls
  beam (idle `scaleX 0.16`) animates 1→1 invisibly on first start because a
  `renderedInlineStyle` recompute resets the transform to base first. Out of
  plan 005's scope; needs its own red test + plan.

- **Wildcard keyframes land at 0, not the current value** (found by plan 007's
  investigation): `controls.start({ x: [0, null] })` from x=64 lands at 0 —
  WAAPI normalization drops the channel — where upstream resolves `null` to the
  value at animation start (64). The settle writer is clean (007's actual
  question), but the live-animation resolution diverges. Low priority; needs
  its own plan against the animation input layer, not `resolveRestingValues`.

## Findings considered and rejected

- **Interrupted-spring velocity loss at composed-writer boundaries** (audit
  gestures finding 07): structurally requires per-key persistent motion
  values — that IS issue #449 (wholesale VisualElement/createAnimationState
  adoption). Patching it inside the current architecture would duplicate that
  work; deliberately not planned here.
- **Missing-key throw in AnimatePresence children** (controls audit caveat):
  upstream silently coerces keyless children to `""`; our explicit throw is a
  deliberate, documented DX strictness choice — not a defect.
- **Hover capability gating via media queries** (gestures audit, verified
  faithful list): additive safeguard consistent with upstream's per-event
  touch filtering; no change needed.
